# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

The **NSA (North Shore Automation) design system**, distributed as a single CSS file via GitHub Pages. There is no build tool, framework, or package manager — it's static HTML + CSS + one small JS file, designed to be served as-is.

Consumers do **not** clone this repo; they load `https://northshoreautomation.github.io/nsa-brand-design/dist/nsa.css` and follow the rules in `CLAUDE.md.snippet.md`. Releases are cut by a single-phase pipeline in `.github/workflows/release.yml` driven by `release:*` labels on PRs to `main`: on every push to `main` the workflow rebuilds `dist/nsa.css` fresh, deploys the site to GitHub Pages, and — when the merging PR carried `release:major|minor|patch` — tags `vX.Y.Z` (notes from `CHANGELOG.md`'s `[Unreleased]` block) and attaches `dist/nsa.css` to a GitHub Release. PR conventions are advisory-gated by `.github/workflows/pr-checks.yml`.

## Architecture

Two source layers, concatenated into one distributable by `build.sh`:

- `tokens.css` — CSS custom properties only: brand colors, neutral ramp, semantic surfaces, status, product accents, radii, 4pt space scale, elevation, type stack/sizes. Also declares `@font-face` for Microgramma (paths relative to repo root: `fonts/...`) and applies the base reset.
- `components.css` — class-based UI (`.btn`, `.input`, `.card`, `.badge`, `.table`, `.ds-shell`/`.ds-side`/`.ds-main`, etc.) built entirely on tokens from `tokens.css`. Never hardcodes colors or spacing.
- `dist/nsa.css` — the **shipped artifact**, generated by `build.sh` and **gitignored**. The release workflow builds it fresh in the runner per push to `main`, includes it in the Pages artifact (so the public Pages URL serves the current build), and on labeled releases attaches it to the GitHub Release. The script prepends a banner with the version (from `$NSA_VERSION` or `git describe --tags`), adds the IBM Plex `@import` from Google Fonts, then concatenates `tokens.css` (with Microgramma URLs rewritten from `fonts/...` → `../fonts/...` so they resolve to the sibling `fonts/` directory under whichever host is serving the CSS) followed by `components.css` unchanged.

The reference site lives under `site/` (`index.html`, `brand.html`, `color.html`, `typography.html`, `space.html`, `components.html`, `patterns.html`, `product-*.html`). Pages load `../tokens.css` and `../components.css` directly (not `dist/nsa.css`) so edits are visible without rebuilding. They share a sidebar injected by `../sidebar.js` — each page sets `<aside data-ds-side data-current="<key>">` and the script populates nav + active state.

Version is resolved at build time, not stored in the repo. `build.sh` reads it from `$NSA_VERSION` (set by the release workflow) or falls back to `git describe --tags --abbrev=0 --match 'v*'` for local dev, or `"dev"` if there are no tags. `sidebar.js` ships with a `__NSA_VERSION__` placeholder; the release workflow substitutes the real version into the deployed Pages copy only, so `main` itself is never churned by release commits. Locally, the sidebar shows `"dev"` via a runtime fallback.

Product theming is a single override layer at the bottom of `tokens.css`: `[data-product="akomi|tsunami|watcher"]` rebinds `--accent`, `--accent-2`, `--accent-soft`. Components that should retheme per-product reference `var(--accent)`, never the raw product token.

## Workflows

**Editing the system.** Modify `tokens.css` and/or `components.css`. Reload any page under `site/` in a browser to verify. No build step needed for the reference site — it loads sources directly.

**Releasing.** Single-phase, label-driven, fully automatic. Nothing is committed back to `main`. The org ruleset `nsa-protect-default-branch` is branch-scoped, so the workflow's tag push is not blocked.

Author flow:
1. Branch from `main`, edit `tokens.css` / `components.css`.
2. Add entries under `## [Unreleased]` in `CHANGELOG.md`.
3. Open a PR.
4. Label the PR with exactly one of:
   - `release:major` — breaking change (X+1.0.0)
   - `release:minor` — new features, backward-compatible (X.Y+1.0)
   - `release:patch` — fixes only (X.Y.Z+1)
   - `release:skip` — merge without cutting a release (but still redeploys Pages)
5. Merge.

On the merge push to `main`, `.github/workflows/release.yml` runs once and does:

1. Looks up the merging PR's `release:*` label via `gh pr list --search "$GITHUB_SHA"`.
2. Computes the new version from the most recent `v*` git tag (no `VERSION` file).
3. Runs `build.sh` with `NSA_VERSION` set to the new version.
4. Stages a `_site/` directory with `tokens.css`, `components.css`, `dist/nsa.css`, `fonts/`, `assets/`, `site/`, `index.html`, and a copy of `sidebar.js` with `__NSA_VERSION__` substituted to the real version.
5. Uploads `_site/` as a Pages artifact and deploys it via `actions/deploy-pages`.
6. **If the label was `release:major|minor|patch`:** extracts the `[Unreleased]` block from `CHANGELOG.md` (fails loudly if empty), creates an annotated tag `vX.Y.Z` with those notes as the tag message, pushes the tag, and runs `gh release create` with `dist/nsa.css` attached.
7. `release:skip` (or no label) deploys Pages but does not tag or release.

`CHANGELOG.md` is **not** auto-rewritten. The `[Unreleased]` block keeps accumulating bullets across releases; humans optionally promote entries under a dated `## [X.Y.Z] — YYYY-MM-DD` heading in normal source PRs when they want a tidy timeline. Skip the promotion if you don't care — the release-notes content lives durably in the annotated tag message and the GitHub Release body.

GitHub Pages must be configured with **Build and deployment → Source: GitHub Actions** (not "Deploy from a branch"). Repo must stay public for Pages.

**Emergency / manual release.** The workflow exposes a `workflow_dispatch` trigger with a `force_version` input. Run it from the Actions tab to publish a specific version without going through a PR — typically only useful if the label-based path is broken. The forced version still goes through the same tag + Release path.

**Adding a new reference page.** Create `site/<name>.html`, link `../tokens.css` + `../components.css` + IBM Plex preconnect/`<link>`, include `<aside class="ds-side" data-ds-side data-current="<key>"></aside>` and `<script src="../sidebar.js"></script>`. Then add a matching `<a data-k="<key>" href="<name>.html">` entry inside `sidebar.js`.

**Adding a new product theme.** Add color tokens to the `Product accents` section of `tokens.css`, add a `[data-product="<name>"]` block at the bottom, create `product-<name>.html`, and add it to `sidebar.js`.

## PR conventions

Every PR to `main` is checked against these conventions by `.github/workflows/pr-checks.yml`. The checks are **advisory** — a failure shows as a red ✕ in the PR's check list but does not block merge. Treat the ✕ as a prompt to look before clicking merge.

1. **Label discipline.** Exactly one `release:*` label per PR:
   - `release:major` — breaking change (X+1.0.0)
   - `release:minor` — new feature, backward-compatible (X.Y+1.0)
   - `release:patch` — fix only (X.Y.Z+1)
   - `release:skip` — merge with no release cut
   Zero labels or multiple `release:*` labels fail the check.

2. **CHANGELOG enforcement.** PRs labeled `release:major`, `release:minor`, or `release:patch` must add at least one new bullet under `## [Unreleased]` in `CHANGELOG.md` vs. `origin/main`. `release:skip` PRs are exempt.

If a check fails, fix the underlying convention rather than waving it through — the rules exist because skipping them is what creates broken-looking releases.

## Rules that constrain edits

These come from `CLAUDE.md.snippet.md` (the rules consumers see) — the system itself must obey them too:

- Never use raw hex for color in components — always tokens. New colors must be added as tokens first.
- No arbitrary px for spacing/radius. Use `--s-1`…`--s-20` and `--r-xs`…`--r-pill`.
- Three font faces only: Microgramma (display, always italic + uppercase + `letter-spacing: .02em` for headlines), IBM Plex Sans (body/UI), IBM Plex Mono (code/metadata/IDs).
- Dark-first. Body is `var(--bg)` (`#0A0F1C`). Single accent is `var(--nsa-signal)` `#5EA1FF` unless `data-product` is set.
- Voice: specific, calm, zero fluff. No exclamation points. No "unlock", "supercharge", "seamless", "next-gen".
- Gradients are reserved for hero moments — not general UI.

## Files

- `CLAUDE.md.snippet.md` — the rules block consumers paste into **their** `CLAUDE.md`. Edit this when the public contract changes; keep it in sync with what `dist/nsa.css` actually provides.
- `DISTRIBUTION.md` — release/tagging mechanics and the consumer URL format.
- `fonts/` — Microgramma TTFs. **Intentionally tracked in git** (see `.gitignore` comment) — required for GitHub Pages to serve them alongside `dist/nsa.css`. Do not add to `.gitignore`.
- `assets/nsa-logo-white.png` — used by `sidebar.js`. Add additional logo variants here.
- `dist/` — generated by `build.sh`; gitignored. Workflow builds fresh per push.
- `tmp/` — scratch, not committed (gitignored).
